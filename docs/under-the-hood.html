01 · BIG PICTURE

Nebula Arena is a top-down dogfight: create a room code, share it, and every pilot’s browser talks directly to every other pilot. The only “cloud” involvement is signaling — a short handshake so peers can find each other. After that, ship positions, bullets, hits, and kills ride encrypted WebRTC data channels peer-to-peer.

Key insight: hosting the static site (Vite build) only serves HTML/JS/CSS. It never sees your shots, positions, or scores. Close the tab and that peer is gone — there’s no backend world state.

02 · WHAT TRYSTERO GIVES US

Trystero wraps the painful parts of WebRTC multiplayer: room namespaces, automatic serialization, action channels, and pluggable strategies for peer discovery (Nostr, BitTorrent trackers, MQTT, Firebase, Supabase, IPFS, custom WS relay).

joinRoom(config, roomId)

Puts this browser into a named room. Anyone with the same appId + roomId can discover each other.

makeAction(name)

Creates a typed pub/sub (or request/response) channel over the mesh. We use one-way messages for game events.

onPeerJoin / onPeerLeave

Presence hooks. We send a hello intro when someone joins so they get our callsign and color.

selfId

Stable local peer ID for this browser session — used as ship owner IDs and bullet prefixes.

How we join

import { joinRoom, selfId } from 'trystero'

const room = joinRoom(
  { appId: 'nebula-arena-p2pgame-v1' },
  roomCode  // e.g. "K7X2MP"
)

const state = room.makeAction('state')
state.send({ x, y, vx, vy, angle, hp, score, alive, … })
state.onMessage = (msg, { peerId }) => { /* interpolate remote ship */ }

03 · SIGNALING: HOW PEERS FIND EACH OTHER

WebRTC cannot open a direct path cold. Peers must exchange SDP (session description) and ICE candidates (possible network paths). That exchange is signaling. Trystero does it for us over Nostr by default.

  1. Room announce Both browsers connect to public Nostr relays and publish encrypted offer/answer payloads on topics derived from appId + roomId.
  2. SDP handshake One peer’s offer is answered by the other. ICE gathers host / srflx / relay candidates (STUN helps discover public addresses).
  3. DTLS + data channel Once ICE succeeds, a secure RTCDataChannel opens. Trystero multiplexes our named actions on it.
  4. Game traffic stays P2P Positions and bullets never go to Nostr again. Relays only helped set up the call.
NAT reality check: some network pairs can’t punch through. Production apps often add a TURN server via Trystero’s turnConfig — data still end-to-end encrypted; TURN only relays ciphertext. Same-machine two-tab testing almost always works.

04 · GAME CHANNELS (ACTIONS)

Nebula Arena defines five lightweight actions. All payloads are plain JSON objects — Trystero serializes and chunks automatically.

Action When Payload (concept)
hello On join / introduce name color score
state ~20 Hz while playing x,y,vx,vy angle hp alive seq
fire Each shot bullet id origin velocity
hit Shooter detects impact targetId damage bulletId
kill Destroy confirmed killer victim names for feed

Why not stream the whole world?

Bandwidth and latency favor small, frequent state plus eventful shots. Each client runs its own physics for local bullets and ships; remotes are visual replicas driven by snapshots.

05 · AUTHORITY & FAIRNESS MODEL

Fully server-authoritative games need a host. We chose a classic P2P pattern that feels good for small arenas:

You own your ship

Only your client decides your thrust, aim, and local position. You broadcast the result at 20 Hz with a sequence number so old packets can’t rewind you.

Shooter-side hits

Your bullets are simulated on every peer, but you detect collisions with remotes and send hit. The victim applies damage (and can ignore duplicates via bullet id).

Interpolation

Remote ships don’t teleport: we blend from last pose → newest snapshot with smoothstep so 20 Hz looks like 60 fps motion.

Trust model

Friends/LAN/demo first. A cheater could lie about hits — competitive ranked play would need a host peer or server. For a serverless toy arena, this is the right tradeoff.

T+0ms
You press fire → local bullet spawns → fire action floods peers
T+~30–80ms
Peers spawn the same bullet vector; everyone sees the shot
T+hit
Your client: distance(bullet, remote) < radius → send hit
T+hit+RTT/2
Victim reduces HP; if HP≤0 → death, respawn timer, killer increments score

06 · CLIENT GAME LOOP

One requestAnimationFrame loop drives everything. Network is event-driven; simulation is frame-driven. 

// each frame (~16ms)
update(dt):
  interpolate remote ships from snapshot A → B
  if dead: countdown respawn
  else: apply WASD thrust, drag, boost, clamp to arena
        aim toward mouse world position
        fire if cooldown ready
  integrate bullets & particles
  local hit-tests for my bullets vs remotes
  every 50ms: broadcast state snapshot

draw():
  camera follow local ship
  nebula backdrop + grid + boundary glow

07 · ENCRYPTION & PRIVACY

  • Signaling: Trystero encrypts SDP with a key derived from appId + room id (optional stronger shared password).
  • Media path: WebRTC data channels use DTLS — eavesdroppers on the path see ciphertext.
  • Room codes: Six-character codes are unguessable enough for casual play; treat them like invite links.
  • No accounts: Callsign is localStorage vanity only; selfId is ephemeral to the session.
